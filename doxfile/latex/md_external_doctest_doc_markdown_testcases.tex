While {\bfseries{doctest}} fully supports the traditional, x\+Unit, style of class-\/based fixtures containing test case methods this is not the preferred style. Instead {\bfseries{doctest}} provides a powerful mechanism for nesting subcases within a test case. For a more detailed discussion and examples see the \href{tutorial.md\#test-cases-and-subcases}{\texttt{ {\bfseries{tutorial}}}}.

Test cases and subcases are very easy to use in practice\+:


\begin{DoxyItemize}
\item {\bfseries{T\+E\+S\+T\+\_\+\+C\+A\+SE($\ast$$\ast$ {\itshape test name} $\ast$$\ast$)}}
\item {\bfseries{S\+U\+B\+C\+A\+SE($\ast$$\ast$ {\itshape subcase name} $\ast$$\ast$)}}
\end{DoxyItemize}

{\itshape test name} and {\itshape subcase name} are free form, quoted, strings. Test names don\textquotesingle{}t have to be unique within the {\bfseries{doctest}} executable. They should also be string literals.

It is possible to write test cases inside of class bodies in C++17 with the help of {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+S\+E\+\_\+\+C\+L\+A\+S\+S()} -\/ used just like {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+S\+E()} -\/ making testing private parts of classes easier.

Keep in mind that even though {\bfseries{doctest}} is \href{faq.md\#is-doctest-thread-aware}{\texttt{ {\bfseries{thread-\/safe}}}} -\/ using subcases has to be done only in the main test runner thread.

Test cases can also be parameterized -\/ see the $\ast$$\ast$documentation$\ast$$\ast$

Test cases and subcases can be filtered through the use of the $\ast$$\ast$command line$\ast$$\ast$\hypertarget{md_external_doctest_doc_markdown_testcases_autotoc_md145}{}\doxysection{B\+D\+D-\/style test cases}\label{md_external_doctest_doc_markdown_testcases_autotoc_md145}
In addition to {\bfseries{doctest}}\textquotesingle{}s take on the classic style of test cases, {\bfseries{doctest}} supports an alternative syntax that allow tests to be written as \char`\"{}executable specifications\char`\"{} (one of the early goals of \href{http://dannorth.net/introducing-bdd/}{\texttt{ Behaviour Driven Development}}). This set of macros map on to {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}s and {\ttfamily S\+U\+B\+C\+A\+SE}s, with a little internal support to make them smoother to work with.


\begin{DoxyItemize}
\item {\bfseries{S\+C\+E\+N\+A\+R\+IO($\ast$$\ast$ {\itshape scenario name} $\ast$$\ast$)}}
\end{DoxyItemize}

This macro maps onto {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE} and works in the same way, except that the test case name will be prefixed by \char`\"{}\+Scenario\+: \char`\"{}


\begin{DoxyItemize}
\item {\bfseries{S\+C\+E\+N\+A\+R\+I\+O\+\_\+\+T\+E\+M\+P\+L\+A\+TE($\ast$$\ast$ {\itshape scenario name}, {\itshape type}, {\itshape list of types} $\ast$$\ast$)}}
\end{DoxyItemize}

This macro maps onto {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+S\+E\+\_\+\+T\+E\+M\+P\+L\+A\+TE} and works in the same way, except that the test case name will be prefixed by \char`\"{}\+Scenario\+: \char`\"{}


\begin{DoxyItemize}
\item {\bfseries{S\+C\+E\+N\+A\+R\+I\+O\+\_\+\+T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+D\+E\+F\+I\+NE($\ast$$\ast$ {\itshape scenario name}, {\itshape type}, {\itshape id} $\ast$$\ast$)}}
\end{DoxyItemize}

This macro maps onto {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+S\+E\+\_\+\+T\+E\+M\+P\+L\+A\+T\+E\+\_\+\+D\+E\+F\+I\+NE} and works in the same way, except that the test case name will be prefixed by \char`\"{}\+Scenario\+: \char`\"{}


\begin{DoxyItemize}
\item {\bfseries{G\+I\+V\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{W\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{T\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

These macros map onto {\ttfamily S\+U\+B\+C\+A\+SE}s except that the subcase names are the \+\_\+something\+\_\+s prefixed by \char`\"{}given\+: \char`\"{}, \char`\"{}when\+: \char`\"{} or \char`\"{}then\+: \char`\"{} respectively.


\begin{DoxyItemize}
\item {\bfseries{A\+N\+D\+\_\+\+W\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\item {\bfseries{A\+N\+D\+\_\+\+T\+H\+EN($\ast$$\ast$ {\itshape something} $\ast$$\ast$)}}
\end{DoxyItemize}

Similar to {\ttfamily W\+H\+EN} and {\ttfamily T\+H\+EN} except that the prefixes start with \char`\"{}and \char`\"{}. These are used to chain {\ttfamily W\+H\+EN}s and {\ttfamily T\+H\+EN}s together.

When any of these macros are used the console reporter recognises them and formats the test case header such that the Givens, Whens and Thens are aligned to aid readability.

Other than the additional prefixes and the formatting in the console reporter these macros behave exactly as {\ttfamily T\+E\+S\+T\+\_\+\+C\+A\+SE}s and {\ttfamily S\+U\+B\+C\+A\+SE}s. As such there is nothing enforcing the correct sequencing of these macros -\/ that\textquotesingle{}s up to the programmer!\hypertarget{md_external_doctest_doc_markdown_testcases_autotoc_md146}{}\doxysection{Test fixtures}\label{md_external_doctest_doc_markdown_testcases_autotoc_md146}
Although {\bfseries{doctest}} allows you to group tests together as subcases within a test case, it can still be convenient, sometimes, to group them using a more traditional test fixture. {\bfseries{doctest}} fully supports this too. You define the test fixture as a simple structure\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{ \{c++\}}
\DoxyCodeLine{class UniqueTestsFixture \{}
\DoxyCodeLine{private:}
\DoxyCodeLine{    static int uniqueID;}
\DoxyCodeLine{protected:}
\DoxyCodeLine{    DBConnection conn;}
\DoxyCodeLine{public:}
\DoxyCodeLine{    UniqueTestsFixture() : conn(DBConnection::createConnection("myDB")) \{\}}
\DoxyCodeLine{protected:}
\DoxyCodeLine{    int getID() \{}
\DoxyCodeLine{        return ++uniqueID;}
\DoxyCodeLine{    \}}
\DoxyCodeLine{\};}
\DoxyCodeLine{}
\DoxyCodeLine{int UniqueTestsFixture::uniqueID = 0;}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_CASE\_FIXTURE(UniqueTestsFixture, "Create Employee/No Name") \{}
\DoxyCodeLine{    REQUIRE\_THROWS(conn.executeSQL("INSERT INTO employee (id, name) VALUES (?, ?)", getID(), ""));}
\DoxyCodeLine{\}}
\DoxyCodeLine{TEST\_CASE\_FIXTURE(UniqueTestsFixture, "Create Employee/Normal") \{}
\DoxyCodeLine{    REQUIRE(conn.executeSQL("INSERT INTO employee (id, name) VALUES (?, ?)", getID(), "Joe Bloggs"));}
\DoxyCodeLine{\}}
\end{DoxyCode}


The two test cases here will create uniquely-\/named derived classes of Unique\+Tests\+Fixture and thus can access the {\ttfamily get\+I\+D()} protected method and {\ttfamily conn} member variables. This ensures that both the test cases are able to create a D\+B\+Connection using the same method (D\+RY principle) and that any ID\textquotesingle{}s created are unique such that the order that tests are executed does not matter.\hypertarget{md_external_doctest_doc_markdown_testcases_autotoc_md147}{}\doxysection{Test suites}\label{md_external_doctest_doc_markdown_testcases_autotoc_md147}
Test cases can be grouped into test suites. This is done with {\ttfamily T\+E\+S\+T\+\_\+\+S\+U\+I\+T\+E()} or {\ttfamily T\+E\+S\+T\+\_\+\+S\+U\+I\+T\+E\+\_\+\+B\+E\+G\+I\+N()} / {\ttfamily T\+E\+S\+T\+\_\+\+S\+U\+I\+T\+E\+\_\+\+E\+N\+D()}.

For example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{ \{c++\}}
\DoxyCodeLine{TEST\_CASE("") \{\} // not part of any test suite}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_SUITE("math") \{}
\DoxyCodeLine{    TEST\_CASE("") \{\} // part of the math test suite}
\DoxyCodeLine{    TEST\_CASE("") \{\} // part of the math test suite}
\DoxyCodeLine{\}}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_SUITE\_BEGIN("utils");}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_CASE("") \{\} // part of the utils test suite}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_SUITE\_END();}
\DoxyCodeLine{}
\DoxyCodeLine{TEST\_CASE("") \{\} // not part of any test suite}
\end{DoxyCode}


Then test cases from specific test suites can be executed with the help of filters -\/ check out the $\ast$$\ast$command line$\ast$$\ast$\hypertarget{md_external_doctest_doc_markdown_testcases_autotoc_md148}{}\doxysection{Decorators}\label{md_external_doctest_doc_markdown_testcases_autotoc_md148}
Test cases can be {\itshape decorated} with additional attributes like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{ \{c++\}}
\DoxyCodeLine{TEST\_CASE("name"}
\DoxyCodeLine{          * doctest::description("shouldn't take more than 500ms")}
\DoxyCodeLine{          * doctest::timeout(0.5)) \{}
\DoxyCodeLine{    // asserts}
\DoxyCodeLine{\}}
\end{DoxyCode}


Multiple decorators can be used at the same time. These are the currently supported decorators\+:


\begin{DoxyItemize}
\item $\ast$$\ast${\ttfamily skip(bool = true)}$\ast$$\ast$ -\/ marks the test case to be skipped from execution -\/ unless the {\ttfamily -\/-\/no-\/skip} option is used
\item $\ast$$\ast${\ttfamily may\+\_\+fail(bool = true)}$\ast$$\ast$ -\/ doesn\textquotesingle{}t fail the test if any given assertion fails (but still reports it) -\/ this can be useful to flag a work-\/in-\/progress, or a known issue that you don\textquotesingle{}t want to immediately fix but still want to track in the your tests
\item $\ast$$\ast${\ttfamily should\+\_\+fail(bool = true)}$\ast$$\ast$ -\/ like $\ast$$\ast${\ttfamily may\+\_\+fail()}$\ast$$\ast$ but fails the test if it passes -\/ his can be useful if you want to be notified of accidental, or third-\/party, fixes
\item $\ast$$\ast${\ttfamily expected\+\_\+failures(int)}$\ast$$\ast$ -\/ defines the number of assertions that are expected to fail within the test case -\/ reported as failure when the number of failed assertions is different than the declared expected number of failures
\item $\ast$$\ast${\ttfamily timeout(double)}$\ast$$\ast$ -\/ fails the test case if its execution exceeds this limit (in seconds) -\/ but doesn\textquotesingle{}t terminate it -\/ that would require subprocess support
\item $\ast$$\ast${\ttfamily test\+\_\+suite(\char`\"{}name\char`\"{})}$\ast$$\ast$ -\/ can be used on test cases to override (or just set) the test suite they are in
\item $\ast$$\ast${\ttfamily description(\char`\"{}text\char`\"{})}$\ast$$\ast$ -\/ a description of the test case
\end{DoxyItemize}

The values that the decorators take are computed while registering the test cases (during global initialization) -\/ before entering {\ttfamily main()} and not just before running them.

Decorators can also be applied to test suite blocks and all test cases in that block inherit them\+:

\DoxyCodeLine{ \{c++\}}

\DoxyCodeLine{        // asserts}
\DoxyCodeLine{    \}}
\DoxyCodeLine{\}}
\DoxyCodeLine{TEST\_SUITE("some TS") \{}
\DoxyCodeLine{    TEST\_CASE("no description even though in the same test suite as the one above") \{}
\DoxyCodeLine{        // asserts}
\DoxyCodeLine{    \}}
\DoxyCodeLine{\}}
\end{DoxyCode}
